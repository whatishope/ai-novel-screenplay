# AI 小说转剧本工具产品设计文档

## 1. 项目背景

本项目面向“AI 小说转剧本工具”作品挑战，目标是利用大模型和结构化数据处理能力，将小说章节内容转换为可编辑、可校验、可导出的 YAML 剧本。

传统小说改编剧本通常需要创作者反复梳理人物、场景、冲突和对白。这个过程耗时较长，且早期草稿容易出现人物遗漏、场景顺序混乱、原文依据不清等问题。本项目希望通过 AI 辅助创作，把小说文本先转化为结构化剧本初稿，帮助创作者更快进入修改和打磨阶段。

## 2. 目标用户

本项目的目标用户主要包括：

- 小说作者：希望将已有小说章节改编为短剧、影视剧、分镜或舞台剧初稿。
- 短剧创作者：需要快速从故事文本中提取可拍摄的场景和对白。
- 编剧学习者：希望理解小说到剧本的结构化改编过程。
- 内容团队：需要批量评估小说文本是否具备改编潜力。

## 3. 用户痛点

小说改编剧本时，用户常遇到以下问题：

- 人物关系和人物目标需要人工整理，工作量大。
- 小说叙事和剧本场景结构不同，直接改写容易混乱。
- 原文中的关键冲突、动作和对白可能在改编时被遗漏。
- 大模型直接生成的剧本文本不够稳定，难以继续编辑和程序化处理。
- 用户需要在生成后继续修改，但普通文本不方便校验、追溯和导出。

## 4. 产品目标

本项目的核心目标是构建一个简单可运行的 AI 辅助剧本创作工具，让用户可以完成从小说文本到结构化剧本初稿的完整流程。

产品目标包括：

- 支持上传至少 3 个章节的小说文本。
- 自动识别章节并拆分内容。
- 提取主要角色、身份描述、性格和目标。
- 根据章节内容规划剧本场景。
- 生成符合 YAML Schema 的结构化剧本。
- 支持在线查看、编辑、校验和导出 YAML。
- 保留原文章节与剧本场景的追溯关系。

## 5. 核心功能

### 5.1 小说文本上传

用户可以上传 `.txt` 小说文本。系统读取 UTF-8 文本内容，返回文件名、文本长度和内容预览，方便用户确认上传结果。

### 5.2 自动章节识别

系统根据章节标题规则识别小说章节，例如“第一章”“第1章”“第一回”“Chapter 1”等。如果无法识别明确章节，系统会按照文本长度自动切分为多个部分，保证后续流程可以继续演示。

### 5.3 角色提取

系统根据章节内容提取主要角色，输出角色 ID、姓名、身份描述、性格、目标和首次出现章节。角色信息用于后续场景规划和剧本生成。

### 5.4 场景规划

系统根据章节和角色列表拆分剧本场景。每个场景包含场景编号、标题、地点、时间、出场人物、剧情摘要和来源章节。

### 5.5 YAML 剧本生成

系统将角色、场景、动作和对白组织为结构化 YAML。相比普通文本，YAML 更便于编辑、校验、导出和后续扩展。

### 5.6 YAML 校验

系统检查 YAML 是否包含必要字段，例如 `metadata`、`characters`、`scenes`，并返回错误和警告，帮助用户发现结构问题。

### 5.7 在线编辑与导出

用户可以在前端页面中查看和编辑 YAML 内容，校验通过后导出为本地 `screenplay.yaml` 文件。

## 6. 技术路线

### 6.1 后端技术

后端采用 Java 17 和 Spring Boot 3.x，负责文件上传、章节识别、AI 调用、YAML 生成和 YAML 校验。

主要技术包括：

- Spring Web：提供 REST API。
- Spring Validation：处理请求参数校验。
- SnakeYAML：解析和生成 YAML。
- Spring AI 或 HTTP Client：调用大模型 API。
- Maven：管理项目依赖和构建。

### 6.2 前端技术

前端采用 Vue 3、Vite 和 Element Plus，提供小说上传、章节展示、生成进度、YAML 预览、在线编辑和导出功能。

主要技术包括：

- Vue 3：构建前端交互界面。
- Vite：提供前端开发和构建能力。
- Element Plus：提供基础 UI 组件。
- Axios：调用后端接口。
- Monaco Editor 或 textarea：编辑 YAML 文本。

### 6.3 大模型调用策略

系统通过统一 LLM 客户端调用大模型。API Key 从环境变量读取，不写死在代码中。如果本地没有配置 API Key，系统返回 mock 数据，保证项目可以在演示环境中稳定运行。

### 6.4 数据存储策略

MVP 阶段不强制使用数据库，优先采用内存或本地临时结果完成演示链路。后续可以扩展 MySQL 保存项目、章节、角色和场景，使用 Redis 保存生成进度和临时任务状态。

## 7. MVP 范围

MVP 阶段聚焦完成一条可演示的主流程：

1. 用户上传小说 `.txt` 文件。
2. 系统读取文本并展示预览。
3. 用户触发章节识别。
4. 系统提取角色并规划场景。
5. 系统生成结构化 YAML 剧本。
6. 用户在线查看、编辑和校验 YAML。
7. 用户导出最终 YAML 文件。

MVP 阶段暂不重点实现用户登录、多项目管理、数据库持久化、复杂权限控制和多人协作功能。

## 8. 后续扩展方向

项目后续可以从以下方向继续增强：

- 单个场景重新生成：用户可以针对某个场景提出修改要求，例如“对白更口语化”或“增加悬疑氛围”。
- 人物一致性检查：检查角色性格、目标和称呼是否前后矛盾。
- 剧情遗漏检查：对比原文章节和剧本场景，提示可能遗漏的关键情节。
- 多种剧本风格：支持短剧、电影、舞台剧、分镜脚本等不同输出格式。
- 自然语言修改剧本：用户用自然语言描述修改需求，系统自动更新 YAML。
- 生成质量报告：输出角色完整度、场景覆盖率、对白比例和追溯完整性等指标。
- 数据持久化：支持保存多个项目和历史生成版本。
- Docker 部署：提供更方便的本地启动和演示环境。
